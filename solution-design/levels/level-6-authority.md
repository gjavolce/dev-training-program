# Level 6 — Authority

**Focus:** Architectural governance, technology strategy, and building the organisation's design capability.

## Which of these scenarios can you handle confidently today?

- When the organisation needs a coherent architecture for a new strategic direction (e.g. expanding into a new market, onboarding a new banking client, or offering a new BaaS product), I can lead the architecture from first principles: domain modelling, system boundary design, data strategy, and platform requirements.
  `domain modelling` `bounded context` `context mapping` `event storming` `system boundary` `data strategy` `BaaS platform` `strategic DDD`
- When the way we make architectural decisions is inconsistent or poorly documented, I can redesign the decision-making process: define when ADRs are required, who needs to be involved for which scopes, and how decisions get revisited when context changes.
  `ADR governance` `decision process` `architectural review board` `RFC process` `design guild` `RACI` `decision log` `revisiting decisions`
- When an audit or regulatory review questions our system design (how data flows, where data lives, what happens in a failure), I can own the response: provide the evidence, identify any genuine gaps, and propose the remediations at an architectural level.
  `PCI-DSS` `SOX` `data flow diagram` `regulatory audit` `data residency` `failure mode documentation` `audit evidence` `gap analysis` `remediation roadmap`
- When we need to assess technical debt across the platform and prioritise what to fix first, I can produce a structured assessment: map the current architecture, identify the highest-risk coupling and inconsistency, and propose a sequenced improvement roadmap.
  `technical debt` `coupling analysis` `architectural risk` `dependency mapping` `improvement roadmap` `strangler fig` `fitness function` `sequenced migration`
- When engineers across the organisation don't have a shared vocabulary for discussing architecture, I can build it: define the standards, run the design guild, and create the reference architectures that teams use as starting points.
  `reference architecture` `design guild` `architectural vocabulary` `pattern library` `architectural standard` `inner source` `community of practice` `golden path`
