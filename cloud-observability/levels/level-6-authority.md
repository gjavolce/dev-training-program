# Level 6 — Authority

**Focus:** Cloud platform strategy, reliability engineering at the organisational level, and technology direction.

## Which of these scenarios can you handle confidently today?

- When the organisation needs to decide its long-term GCP strategy (which managed services to use, what to run on GKE vs serverless vs managed, how to manage multi-region), I can lead the analysis and produce a technology direction document that engineering leadership can act on.
  `GKE vs Cloud Run vs GKE Autopilot` `multi-region architecture` `managed services trade-offs` `GCP service selection` `technology radar` `architecture decision record` `long-term cost model` `vendor strategy`

- When we need to validate that our disaster recovery procedures actually work (not just that they're documented), I can design and run the DR drill: define success criteria, coordinate the participating teams, execute the failover, measure the results, and produce a gap analysis.
  `DR drill` `Cloud SQL failover` `RTO / RPO` `GKE multi-region failover` `DR runbook` `success criteria` `gap analysis` `chaos engineering`

- When the organisation's Cloud spend is being reviewed at the executive level, I can own the cost narrative: explain the main drivers, identify the highest-leverage optimisation opportunities, and present a credible roadmap for cost management as we scale.
  `GCP Billing export` `BigQuery cost dashboard` `committed use discounts` `sustained use discounts` `FinOps` `cost roadmap` `unit economics` `executive cost narrative`

- When a regulatory audit requires evidence of how we detect, alert on, and respond to security events in our cloud environment, I can produce that evidence: describe the detection mechanisms, the alerting chain, the response procedures, and the audit log retention.
  `Cloud Audit Logs` `Cloud IAM audit trail` `Cloud KMS key usage` `Security Command Center` `log retention policy` `alert evidence` `PCI-DSS / SOX compliance` `audit log export`

- When we need to build a platform capability that will benefit all engineering teams (e.g. a standard observability stack, a shared incident management process, a Terraform module library), I can lead the design and rollout — ensuring it's adopted because it's genuinely useful, not because it's mandated.
  `Terraform module registry` `shared observability stack` `platform capability design` `inner-source adoption` `OTEL collector deployment` `Grafana provisioning` `platform engineering` `developer experience`
